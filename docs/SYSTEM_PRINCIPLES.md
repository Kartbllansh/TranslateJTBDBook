# Принципы системы перевода книг

## Назначение

Этот репозиторий постепенно превращается из проекта перевода одной книги в локальную систему для перевода книг на русский язык. Система должна брать PDF на входе, извлекать структуру книги, помогать перевести и вычитать текст, а затем публиковать удобную HTML-читалку.

Главный принцип: каждый этап должен оставлять понятные файлы на диске. Пользователь должен иметь возможность остановиться, поправить текст вручную и продолжить пайплайн без потери работы.

## Целевой поток

1. **Импорт PDF**
   Пользователь добавляет PDF книги в проект.

2. **Извлечение**
   `opendataloader-pdf` извлекает из PDF Markdown и JSON. Markdown нужен для текста, JSON нужен для структуры, страниц, координат блоков и будущей проверки качества.

3. **Нормализация книги**
   Наш слой очищает результат: убирает номера страниц, headers/footers, склеивает переносы, выделяет главы, приложения и примечания.

4. **Перевод**
   Текст переводится по главам и чанкам. Система использует терминологическую базу, сохраняет прогресс и не перезаписывает ручные правки без явного действия.

5. **Редактура**
   Пользователь сравнивает оригинал и перевод, исправляет ошибки, проверяет термины и стиль.

6. **Публикация**
   Система собирает reader-сайт и автономный HTML-файл, которым можно делиться.

## Роли компонентов

### `opendataloader-pdf`

**Роль**: извлечь структурированные данные из PDF.

**Вход**: PDF-файл или папка PDF-файлов.

**Выход**:
- Markdown для дальнейшей разбивки и перевода.
- JSON для структуры, страниц, типов блоков и координат.

**Что не делает**:
- Не принимает переводческие решения.
- Не знает правила конкретной книги.
- Не отвечает за стиль русского текста.
- Не собирает финальную читалку.

### Book pipeline

**Роль**: связать извлечение, нормализацию, перевод, редактуру и публикацию.

Планируемые папки:

```text
projects/
  <book-id>/
    source/        # результат извлечения из PDF
    chapters/      # очищенные главы оригинала
    translated/    # русский перевод
    review/        # отчеты, замечания, спорные места
    dist/          # готовые файлы для публикации
    metadata.json  # название, автор, язык, структура
```

### Reader

**Роль**: дать удобное чтение готового перевода.

Текущий reader уже умеет:
- строить оглавление;
- искать по книге;
- менять тему и размер текста;
- собираться в один автономный HTML-файл.

## Текущий первый шаг

В репозиторий добавлен CLI-слой:

```bash
.venv/bin/python -m book_pipeline.extract_opendataloader WCAKC.pdf -o projects/wcakc/source
```

По умолчанию он просит у `opendataloader-pdf` форматы:

```text
markdown,json
```

Если нужен CLI-режим:

```bash
.venv/bin/python -m book_pipeline.extract_opendataloader WCAKC.pdf -o projects/wcakc/source --use-cli
```

Полезные опции для книг:

```bash
.venv/bin/python -m book_pipeline.extract_opendataloader WCAKC.pdf \
  -o projects/wcakc/source \
  --format markdown,json \
  --image-output external \
  --use-struct-tree
```

Для быстрой проверки нескольких страниц:

```bash
.venv/bin/python -m book_pipeline.extract_opendataloader WCAKC.pdf \
  -o projects/wcakc/source-smoke \
  --pages "1-5"
```

## Prerequisites

`opendataloader-pdf` требует Java 11+ и Python 3.10+.

Установка Python-зависимости:

```bash
python3 -m venv .venv
.venv/bin/python -m pip install -r requirements.txt
```

Проверка Java:

```bash
java -version
```

## Принципы проектирования

### 1. Файлы важнее скрытого состояния

Все важные результаты должны лежать в файловой структуре проекта. Не полагаемся только на базу данных или состояние веб-интерфейса.

### 2. Каждый этап можно повторить

Импорт, извлечение, нормализация, перевод и публикация должны быть повторяемыми командами. Если этап перезапускается, он не должен случайно уничтожать ручные правки.

### 3. Ручная редактура считается нормой

Цель системы не “перевести без участия человека”, а ускорить качественный перевод. Поэтому интерфейс должен поддерживать исправления, комментарии и повторную сборку.

### 4. Терминология централизована

Ключевые термины живут в `TERMINOLOGY.md`. Переводчик и будущие автоматические проверки должны сверяться с этим файлом.

### 5. Оригинал и перевод связаны

Каждая переведенная глава должна иметь ссылку на исходную главу. В будущем JSON с координатами позволит показывать страницу и место в PDF.

### 6. Публикация отделена от перевода

Reader не должен знать, как именно был сделан перевод. Он получает готовые Markdown-файлы и собирает удобный сайт или автономный HTML.

## Ближайшие решения

Нужно отдельно спроектировать:

- формат `metadata.json`;
- правила автоматической разбивки на главы;
- способ хранения чанков и статусов перевода;
- интерфейс сравнения оригинала и перевода;
- команды для сборки reader из любого проекта, а не только из текущей папки `translated/`.

Подробная рабочая спецификация репозитория находится в `docs/REPOSITORY_SPEC.md`.

## Открытые вопросы

- Поддерживаем ли только PDF или сразу EPUB/DOCX?
- Где хранить изображения из книги?
- Нужно ли сохранять постраничную привязку в финальной читалке?
- Как отмечать места, где переводчик сознательно отошел от буквального перевода?
- Должен ли standalone HTML включать оригинал рядом с переводом или только готовый русский текст?
